# create-skill 元技能

## 核心任务

根据需求创建或重构 Claude Code Skill 文件所在的目录结构及其内容。

## 适用场景

| 任务类型        | 所需规则                                 |
|-------------|--------------------------------------|
| 创建通用 Skill  | 仅本文件                                 |
| 重构通用 Skill  | 本文件 + **`重构规则.md`**                  |
| 创建代码类 Skill | 本文件 + **`代码类约束.md`**                 |
| 重构代码类 Skill | 本文件 + **`重构规则.md`** + **`代码类约束.md`** |

所有依赖文件在本目录下，使用相对路径读取。

---

## 示例的泛化约束

**所有示例必须使用抽象占位符，禁止直接或间接引用真实 Skill 名、文件名、目录名。**

| 约束         | 禁止                                           | 必须                                   |
|------------|----------------------------------------------|--------------------------------------|
| Skill 名称示例 | `read-config`, `build-image`, `deploy`       | `foo-bar`, `baz`, `<name>`           |
| 文件路径示例     | `create-pyi/SKILL.md`, `config-reader/工具.py` | `<skill-dir>/SKILL.md`               |
| 命令参数示例     | `--format pyi`, `--type python`              | `--format <format>`, `--type <type>` |
| 工具脚本名      | `解析器.py`, `_reader.py`, `extract.py`         | `工具.py`, `<tool-name>.py`            |
| 列举真实 Skill | "create-pyi 负责存根, write-docstring 负责文档"      | "A Skill 负责存根, B Skill 负责文档"         |

判断标准：如果示例中的名称能让人联想到某个已存在的实体（Skill、文件、格式、语言），就不够泛化。**读示例的人应该只理解规则本身，不带走任何具体名称。
**

## Skill 命名规则

**必须**遵循 `<动词>` 或 `<动词>-<名词>` 格式，英文 **kebab-case**。不满足此规则的 Skill 名在自动加载时被忽略。

```
# 正确
<动词>-<名词>    例: foo-bar
<动词>           例: baz

# 错误
<名词>-<动词>    名词开头, 不满足动词优先
<名词>-<名词>    无动词
<任意>-<任意>    无动词
```

命名在前件 `name` 字段和目录名中各出现一次，**两者必须完全一致**。

## Skill 目录结构

```
.claude/skills/
└── <name>/
    ├── SKILL.md          # 主文件, 必含 frontmatter
    ├── 规则集.md          # 依赖文件 (可选, 无 frontmatter)
    └── 工具.py            # 工具脚本 (可选)
```

SKILL.md 的前件格式：

```markdown
---
name: <name>        # 动词-名词, kebab-case
description: <触发描述>
version: <语义化版本号>
---
```

## 设计原则

### 1. 关注点分离

一个 Skill 只封装一个独立领域。当发现一个 Skill 涉及两个不共享规则体系的领域时，拆分为多个 Skill。

判断标准：如果 Skill 在输出时可能需要引用另一个 Skill 的规则，就应该拆分。

```
好: 结构变换 Skill + 文档编写 Skill (各管一个领域)
不好: 一个 Skill 同时管结构变换和文档编写
```

### 2. 规则驱动，非模板驱动

Skill 应定义**规则和边界条件**，而非提供可整体复制的完整模板。完整格式参考只出现一次。

```
好: 规则 + 最小示例 (演示单条边界)
不好: 完整成品示例 (容易被整段复制)
```

### 3. 职责边界清晰

Skill **只规定产出**，不规定如何修改用户已有文件。

```
好: 产出: 筛选后的条目, 不修改原始数据
不好: 对原始数据执行清洗和格式转换
```

### 4. 版本号语义

| 版本          | 何时升级              |
|-------------|-------------------|
| 主版本 (x.0.0) | 规则体系重构, 不兼容旧行为    |
| 次版本 (1.x.0) | 新增规则或章节, 向后兼容     |
| 修订版 (1.0.x) | 修正现有规则的表述、示例或边界条件 |

### 5. 质量检查清单

每个 Skill 必须以检查清单结尾，每条检查项是**可验证的**。

```
好: - [ ] 空输入时跳过处理
不好: - [ ] 处理结果合理
```

## 章节组织规范

### 核心任务

一句话说明该 Skill 做什么。不要超过 3 条。如果超过 3 条，考虑拆分 Skill。

### 规则章节

按逻辑分组，每组一个二级标题。规则用列表或表格陈述。

规则描述中：

- 使用 **粗体** 标记关键结论或强制性要求
- 避免模糊量词 ("适当的", "合理的")
- 引用其他 Skill 或本目录下的依赖文件时，使用加粗并标注相对路径

### 示例

每个示例前必须有规则陈述。示例内容应聚焦于演示该条规则，不包含无关信息。

**必须**遵守示例的泛化约束——所有名称使用抽象占位符。见第一节。

### 依赖文件

当 SKILL.md 的内容膨胀到影响阅读，或部分规则只被特定场景引用时，可以拆分到依赖文件中。

依赖文件的组织规则：

- 放在 SKILL.md 同目录下，不单独注册为 Skill
- **不含 frontmatter**, 直接以标题开头写规则内容
- SKILL.md 中使用**必须**或**按需**标注引用条件和相对路径

```
## 适用场景

### 场景 A
**必须**同时读取本目录下的 `依赖文件A.md`。

### 场景 B
**必须**同时读取本目录下的 `依赖文件B.md`。
```

### 工具脚本

当 Skill 需要解析、生成或转换文件时，可以附带工具脚本。

工具脚本的规范：

- 放在 SKILL.md 同目录下
- 语言不限, 但必须自包含（单文件，无外部依赖）或在注释中注明依赖
- SKILL.md 中说明如何调用、传入什么参数、期望什么输出
- **必须**包含适当的错误处理和退出码

命名规范：**不体现具体领域**。使用 `工具.py`, `<name>.<ext>`, `脚本.py` 等抽象名，不反映其实际功能。

调用方式：AI 通过 Bash 工具执行，SKILL.md 中给出精确的命令示例。

```markdown
### 工具脚本

使用本目录下的 `工具.py` 提取信息:
bash <skill-dir>/工具.py <输入文件> --mode <模式>
输出: 每行一个条目。
```

### 输出格式

如果 Skill 涉及生成多项产出，用编号列表明确顺序和格式。

### 质量检查清单

使用 `- [ ] ` 格式的无序列表。按检查对象分组。

## 创建流程

1. **确定 Skill 名称** — 按 `<动词>-<名词>` 英文 kebab-case 拟定，不满足则退回重拟
2. **理解领域边界** — 确定该 Skill 要解决的单一问题，明确不做的事
3. **设计规则体系** — 列出所有规则，确保不重叠、不遗漏
4. **评估是否需要依赖文件或工具脚本** — 规则过多时拆分为依赖文件；需要自动处理时附带工具脚本。**工具脚本名必须使用抽象名**
5. **编写草稿** — 按结构组织。**目录名、frontmatter `name`、标题中的 `# <name>` 三者完全一致**
6. **检查示例泛化性** — 逐条审核所有示例，替换任何非抽象名
7. **添加检查清单** — 确保清单可验证

---

## 质量检查清单

### 名称与结构

- [ ] Skill 名称符合 `<动词>` 或 `<动词>-<名词>` 英文 kebab-case 格式
- [ ] 目录名与 frontmatter `name` 完全一致
- [ ] 工具脚本名使用抽象名（不体现具体领域）

### 示例

- [ ] 所有示例的 Skill 名使用抽象占位符 (`foo-bar`, `<name>`)，无真实 Skill 名
- [ ] 所有示例的文件路径使用抽象占位符，无真实路径
- [ ] 所有示例的命令参数使用抽象占位符，无真实格式/类型名
- [ ] 所有示例的工具脚本名使用抽象占位符，无真实文件名

### 规则完整性

- [ ] Skill 只封装一个独立领域（关注点分离）
- [ ] 规则驱动而非模板驱动（完整模板最多一个）
- [ ] 各章节职责边界清晰，不越界规定操作
- [ ] 质量检查清单可验证
- [ ] 描述能匹配到正确的触发场景
- [ ] 依赖文件的引用路径准确且可读
- [ ] 依赖文件不含 frontmatter
- [ ] 工具脚本自包含或注明外部依赖
- [ ] 工具脚本的调用方式和预期输出已在 SKILL.md 中说明
- [ ] 自身示例不违反自身定义的规则
