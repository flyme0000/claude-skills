# 代码审查规则

仅通过 SKILL.md 引用，供 Claude 在对 staged diff 执行审查时遵循。

**必须**同时读取本目录下的 `review_config.yaml` 和 `ruff.toml`（机械预检规则配置）。

---

## 两阶段审查流程

对 staged diff 依次执行机械预检和 AI 审查，合并结果后输出。

### 阶段一：机械预检

先输出描述：`==> 阶段一：机械预检（Ruff）`

然后用 review.py 进行机械预检，同时保存 JSON 供后续标注使用：

```bash
# 1. 机械预检，输出报告 + 保存 JSON
python <skill-dir>/review.py --json /tmp/mech_findings.json
```

- 基于 review_config.yaml 配置，Ruff 规则选择见 ruff.toml
- 只扫描 Python 文件的新增行
- 输出格式：`文件路径:行号:级别: 分类: 描述`
- 可选 `--json` 输出机械检查结果，供 annotate.py 使用
- 退出码：存在 error/critical 时返回 1，否则返回 0
- **默认只输出报告，不修改文件**

### 阶段二：AI 审查

先输出描述：`==> 阶段二：AI 审查（逻辑 / 完整性 / 安全 / 一致性 / 架构）`

然后按下方[审查维度](#审查维度)对 staged diff 逐一检查。

**不因机械预检已通过而跳过任何维度**。AI 审查可以发现机械规则无法覆盖的逻辑问题、完整性问题和架构问题。

### 结果合并

先输出描述：`==> 合并结果`

然后执行以下步骤：

1. 先执行 `python <skill-dir>/review.py` 拿到机械检查结果
2. 再按审查维度清单检查 diff
3. 两份结果**合并去重**后输出给用户
4. 存在 error/critical 则阻断提交

对于机械预检已发现的问题，AI 审查可以：

- **确认**该问题的严重性
- **降级**（如在序列化工具类中 `pickle.loads` 属于正常使用），说明理由

### 标记注释

机械预检和 AI 审查中**至少有一个发现问题**时，询问用户：

```
是否在源文件中插入 TODO 标记注释？(y/N)
```

若两者均未发现问题，跳过此步骤。

仅当用户确认后执行。

先输出描述：`==> 写入 TODO 标记注释`

然后将 AI 审查发现写入临时 JSON 文件，与机械预检结果合并后标注：

```bash
# 1. 将 AI 审查发现写入临时文件
cat > /tmp/ai_findings.json << 'EOF'
[
  {"filepath": "read-code/read.py", "lineno": 234, "level": "info", "category": "逻辑正确性", "description": "assert 报错信息不友好，建议改为 ValueError"}
]
EOF

# 2. 用 annotate.py 合并标注（机械 + AI）
python <skill-dir>/annotate.py --mech /tmp/mech_findings.json --ai /tmp/ai_findings.json

# 3. 清理临时文件
rm /tmp/mech_findings.json /tmp/ai_findings.json
```

AI 发现 JSON 格式说明：

| 字段          | 类型     | 说明                                    |
|-------------|--------|---------------------------------------|
| filepath    | string | 文件路径（必须，用于定位写入哪个文件）                   |
| lineno      | int    | 行号，0 表示非行级问题（跳过标注）                    |
| level       | string | 级别: error / critical / warning / info |
| category    | string | 分类: 如 "逻辑正确性", "不安全代码"                |
| description | string | 问题描述                                  |

- lineno 为 0 的条目不会被写入文件，仅输出报告
- 重复的 lineno 会在同一文件内从上往下依次插入

标记注释格式（由 annotate.py 写入）：

```python
# TODO)) [级别] @review 分类: 描述
# TODO)) [error] @review 不安全代码: verify=False 跳过证书验证
```

标记**随代码入库**，进入版本控制。修复问题后由开发者手动删除。

### both 模式

当 SKILL.md 中指定 `both` 模式时：

1. 执行机械预检
2. 执行 AI 审查
3. 若存在 error/critical，**中断**，提示用户修复后重试
4. 若有发现问题，用 保存机械结果，再询问是否写入 TODO 标记注释
5. 继续生成提交信息

## 审查维度

对 staged diff 全量分析，按以下维度逐一检查。

### 1. 逻辑正确性

检查变更引入的逻辑是否正确：

- 新增的分支/条件是否有遗漏情况
- 异常是否被静默吞掉（`except: pass`、空 except）
- off-by-one、边界值处理
- 并发场景下的竞态条件
- 递归有无终止条件，深度是否可控
- 类型假设是否匹配实际类型

### 2. 改动完整性

检查变更是否完整，有无遗漏配套修改：

- 新增/修改配置项 → 配置文件是否同步更新
- 新增字段/参数 → 调用方是否已更新
- 修改数据库模型 → migration 是否包含
- 新增 API 端点 → 路由注册、序列化、文档是否配套
- 删减功能 → 相关引用是否已清理

### 3. 上下文安全

对机械预检的结果进行二次判断，区分误报和真问题：

- `print()` — 是临时调试还是在 CLI 工具的正式输出
- `eval()` — 是在 DSL 解析器中的正常使用还是用户输入直接传入
- `time.sleep()` — 是生产主路径的固定延时还是重试/测试中的合理使用
- `subprocess.run()` — 命令是否可控，shell=True 是否必要
- 密钥硬编码 — 是否在测试/示例代码中
- 同时关注 diff 中被**删除**的行：是否删掉了安全措施（`verify=True`、输入验证等）

### 4. Diff 一致性

检查变更是否符合一次 commit 的预期粒度：

- 是否在同一个 commit 中混入了不相关的改动（如同时修 bug 和改配置格式）
- 若有混淆之处，在 body 中按变化类型分组列出

### 5. 架构影响

检查变更对现有代码结构的影响：

- 是否引入循环依赖
- 是否破坏原有接口签名（向后兼容性）
- 是否在错误抽象层实现功能（如业务逻辑写在 view/controller 层）
- 是否过早抽象（为一个假设的未来需求添加框架）

## 评估输出

每个发现问题按以下级别标记：

| 级别       | 含义             | 是否阻断 |
|----------|----------------|------|
| error    | 确定有问题的代码，必须修复  | 阻断   |
| critical | 高度可疑，几乎可以确定有问题 | 阻断   |
| warning  | 潜在隐患，建议修复      | 不阻断  |
| info     | 仅供参考           | 不阻断  |

输出格式（与机械预检一致）：

```
文件路径:行号:级别: 分类: 描述
```
